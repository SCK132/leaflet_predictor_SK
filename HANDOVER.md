# CUSF / SondeHub Landing Predictor (Japanese Localized & Enhanced Version)

## 1. プロジェクト概要
本プロジェクトは、[ProjectHorus/leaflet_predictor](https://github.com/projecthorus/leaflet_predictor) をベースに、WASA（早稲田大学宇宙航空研究会）の運用に合わせて日本語化および機能拡張を行ったものです。

主な目的は、気球（Radio Sonde）の飛行経路および着地位置予測を、**日本語UI**かつ**JST（日本標準時）**ベースで直感的に行えるようにすること、および運用効率化のための独自機能を追加することです。

## 2. 主な変更点と追加機能

### 2.1 日本語化 (Localization)
- **UIの全日本語化**: `index.html` 内のラベル、ボタン、説明文を英語から日本語に翻訳しました。
- **メッセージの日本語化**: エラーメッセージやステータス表示（`pred-new.js` 等）を日本語に変更しました。

### 2.2 JST (日本標準時) 対応
- **デフォルト日時**: ページ読み込み時に、現在時刻(UTC)に9時間を加算し、JSTとしてフォームに初期入力するように変更しました。
- **Moment.js Timezone**: 日時処理の内部ロジックを `moment.utcOffset(9 * 60)` を使用するようにリファクタリングし、より堅牢なJST対応を行いました。
- **入力/表示**:
  - 予測時刻入力フォームは JST として扱われます。
  - マーカー（Launch/Burst/Landing）のツールチップやポップアップに表示される時刻も「XX:XX JST」形式で表示されます。
- **URL共有**: 生成される共有URLもJST基準のパラメータを含むように調整されています（内部処理ではUTC変換）。

### 2.3 落下位置一覧リスト (Feature: Landing Position List)
- 予測を実行するたびに、画面右側のパネル（シナリオ情報下）に以下の履歴がリストとして追記されます。
  - **打ち上げ場所**: 選択したサイト名
  - **打ち上げ時刻**: JST形式
  - **落下場所**: 緯度経度（クリックで地図をパン）
  - **海陸判定結果**: 自動判定（後述）

### 2.4 海陸判定機能 (Feature: Hybrid Land/Sea Discrimination)
  1. **ローカルGeoJSON (即時判定)**: `data/land_japan_raw.geojson` に含まれる日本の陸地ポリゴンに対して、**Ray-Casting アルゴリズム**によるポイント・イン・ポリゴン判定を実施。バウンディングボックスによる高速リジェクト付き。明確に陸と判定された場合は内陸水域チェックへ進みます。
  2. **BigDataCloud API**: ローカル判定で海（ポリゴン外）と判定された場合、または沿岸部の場合にAPIで確認。`countryCode` の有無と `localityInfo` 内の海関連キーワード（Sea, Ocean, Bay等）をチェック。
  3. **Overpass API (内陸水域チェック)**: 陸と判定された場合でも、50m圏内の水域（湖沼・河川）をOverpass APIで確認し、琵琶湖・霞ヶ浦等の内陸水域への着水を検知します。
  4. **フォールバック**: API失敗時はローカルGeoJSON結果をフォールバックとして使用。

- **海陸判定の高速化と精度再強化** (`pred-new.js`, `launch-window.js`):
  - `classifyLandSeaAt()` に判定キャッシュと沿岸モンテカルロ近似を追加し、沿岸付近の判定精度を上げつつ local 実行時の再利用性を改善。
  - local の放球NG再表示では、スライダー移動時に再計算せず、選択時刻のフォーム値と結果表示を再同期するように修正。

- **UI文言の整理** (`index.html`, `pred-new.js`):
  - 主要な表示文言から絵文字を除去し、見出しとトースト文言をテキストベースへ統一。

### 2.5 過去の履歴マーカーの維持
- 予測を再実行しても、過去の予測着地点マーカーを地図上に残すように変更しました（`landing_history_markers`）。
- これにより、パラメータを微調整した際の落下地点のばらつきを視覚的に確認できます。

### 2.6 南レク一括計算 (Batch Simulation)
- 「南レク一括計算」ボタンを追加しました。
- `sites.json` に定義された複数の打ち上げ場所（南レク南楽園、津島プレーランド等）に対して、連続して予測を実行する機能です。
- バッチリストは `pred-new.js` 内の `batchSites` 配列で管理されています。

### 2.7 過去日時シミュレーション対応 (Task B Support)
- **過去日時の制限解除**: フロントエンドの「12時間前まで」という制限を緩和し、任意の過去日時を入力可能にしました。
- **API接続先切り替え**: 打ち上げ設定カードに「API接続先」セレクタを追加しました。
  - **SondeHub (Public)**: デフォルト。通常の予測に使用。
  - **Localhost (Docker)**: 過去の気象データを用いてローカルでTawhiriを動かす場合に使用。`cors-proxy.js` 経由でDockerに接続。
  - **カスタム**: 任意のURLを指定可能。

### 2.8 実測ログ重ね合わせ (Log Overlay)
- **CSVアップロード**: シナリオ情報パネルに、CSVファイル (`lat,lon,alt`) をアップロードする機能を追加しました。
- **地図表示**: アップロードされたログは赤色の破線で地図上に描画され、予測経路（黒）との比較が容易になります。
- **ヘッダー自動検知**: `lat`, `lon`, `alt` 等のキーワードをヘッダー行から自動探索。ヘッダーが無い場合はデータ列の数値範囲から列を推測。
- **外れ値除去**: GPS初期化直後の異常値（0,0付近）や中央値から離れた点を自動除外。

### 2.9 UI改善
- **日付入力**: 年・月・日の順序を日本式 (Y-M-D) に変更し、モバイル端末での表示幅を最適化しました。
- **高度グラフ**: デスクトップ版では地図左下に配置し、視界を遮らないようにしました。「高度グラフを表示/非表示」ボタンで切り替え可能です。
- **ボタン配置**: KMLボタンとShareボタンの間にスペースを設け、誤操作を防止しました。
- **UIの洗練化 (Sophisticated Refresh) [NEW]**:
  - **絵文字の完全排除**: 「安っぽい」印象を与える絵文字をUIからすべて削除し、クリーンなテキストベースのUIへ移行しました。
  - **オリジナル・テーマトグル**: ダークモード切替ボタンを、絵文字（🌙/☀️）からインラインSVGアイコン（太陽/月）を使用したカスタムトグルに刷新。切替時に滑らかな回転・フェードアニメーションを導入しました。
  - **SVGトーストアイコン**: 通知（Success/Warning/Error/Info）のアイコンをモダンなフラットデザインのSVGに置き換えました。
  - **テキストの最適化**: サイドバータブのテキストを `SETTINGS` / `RESULTS` のように大文字化・レタースペーシング調整を行い、よりプロフェッショナルな外観に整えました。

### 2.10 アルゴリズムとシミュレーションモードの強化
- **正規分布アンサンブル (Gaussian Ensemble)**: アンサンブル予測のパラメータ変動（上昇速度、破裂高度等）を、単純な一様乱数から**正規分布（ガウス分布、Box-Muller変換）**に変更しました。標準偏差を約5%に設定し、より現実的な確率分布を持つ着地点のばらつきをシミュレーションします。
- **落下のみモード (Fall Only Mode)**: 飛行プロファイルに「落下のみ (Fall Only)」を追加しました。打ち上げ高度から即座に落下を開始（破裂高度＝打ち上げ高度として処理）するモードで、ドローン投下等のシミュレーションに利用できます。このモード選択時は、上昇速度や破裂高度の入力欄が無効化されます。

### 2.11 13バリアント・アンサンブル (Ehime Balloon Experiment Mode)
- **Falling-position-simulator2025からのマージ**: この機能のベースとなる変動ロジックと閾値（上昇速度±1 m/s、下降速度±3 m/s、破裂高度+10%/-20%）は、旧版である `Falling-position-simulator2025` の仕様に完全に準拠しています。
- **決定論的13パターンシミュレーション**: 上記マージンに従って13パターンの予測を一度に実行します。
  - 各パラメータの単独変動（6パターン）+ 組み合わせ変動（6パターン）+ 基準 = 計13パターン
- **統計情報パネル** (`#ensemble_stats_panel`): 実行進捗、平均着地点、最大偏差(km)、海陸判定割合を表示。（UIは新版を維持）
- **コンベックスハル (凸包) 描画**: Turf.js を使用して着地点群の凸包を地図上に描画し、予測の不確実性エリアを可視化。
- **結果テーブル**: デスクトップではHTML表、モバイルではカード形式で各バリアントの詳細（変動内容、着地座標、使用パラメータ、飛行時間、海陸判定）を表示。
- **各バリアントの色分け表示**: 13色のカラーパレットで経路・着地マーカーを描画。基準(BASE)は太い黒線で強調。

### 2.12 風速プロファイルグラフ [NEW]
- **高度別水平風速グラフ** (`pred-chart.js`): Tawhiri APIの予測結果から隣接する2点間の水平移動速度をHaversine距離で算出し、高度に対する風速散布図（Chart.js scatter plot）を描画します。
- 「風速グラフを表示/非表示」ボタンで切り替え可能。

### 2.13 モバイルUI対応 (Responsive Mobile UI) [NEW]
- **ボトムナビゲーションバー** (`mobile_ui.js`, `mobile.css`): 768px以下の画面幅で、画面下部に固定ナビゲーションバーを表示。「打ち上げ」「結果」「愛媛」「計算機」のパネル切り替えボタンを配置。
- **パネルスライドアップ**: 各パネルは画面下部からスライドアップするシート形式で表示。複数パネルの排他制御もJSで管理。
- **コンパクトモード**: モバイルではフォーム入力欄のスペースを圧縮し、画面を有効活用する `compact` クラスを自動適用。
- **レスポンシブ・アンサンブル結果**: 600px以下ではテーブルの代わりにカード形式UIを自動表示。
- **タッチフレンドリー**: 入力要素のフォントサイズを16pxに拡大し、iOS Safariのズーム問題を回避。

### 2.14 PWA対応 (Progressive Web App) [NEW]
- **Service Worker** (`sw.js`): 静的ファイルとマップタイルのキャッシュにより、オフラインまたは低速ネットワーク環境でも基本的なUI表示が可能。
  - **静的キャッシュ**: `predictor-cache-v7` (HTML, CSS, JS, GeoJSON, アイコン等)
  - **デザインシステム（Apple HIG準拠）**:
  - `predictor.css` にApple Human Interface Guidelinesに基づく厳密なデザイントークン（セマンティクスに基づく変数化、マテリアル背景のぼかし効果、SF Proや角丸指定）を導入。
  - `#input_form` や `#ensemble_stats_panel` 等のパネルUIに `backdrop-filter` とシャドウを適用し、Z軸上のDepthを表現。
  - すべてのインタラクティブ要素（ボタン、入力欄）に最低44pxのタッチターゲットを確保し、屋外でのフィールド運用時の操作性を完全確保。
  - jQuery UIボタンを排し、全てCSSによる標準化された `.btn-primary` 等のクラスに置換。絵文字を削除しクリーンなUIを実現。
- **海陸判定へのモンテカルロサンプリング導入**:
  - 単純な13バリアントの落下軌跡の端点判定から進化させ、予測地点群の平均と共分散を計算。
  - 正規乱数（Box-Muller変換およびコレスキー分解）を用いて**1000個の仮想落下地点**を瞬時にローカル生成し、局所的な「半島飛び越え」などのミスキャストを防ぐ高次元な海陸確率を算出。
- **Webマニフェスト** (`manifest.json`): ホーム画面追加対応。`standalone` モードでブラウザUI無しで起動可能。

### 2.15 コラボレーション機能 (Collaboration) [NEW]
- **URL共有** (`pred-collaborate.js`): 現在の予測パラメータ（API設定含む）を含んだURLをクリップボードにコピー。
- **画像エクスポート**: `html2canvas` を使用して、現在のマップビュー（UI含む）をPNG画像として保存。ファイル名には日時が自動付与。

### 2.16 アンサンブル結果エクスポート (Ensemble Export) [NEW]
- **CSVエクスポート** (`phase4-features.js` → `exportEnsembleCSV()`): 13バリアント結果をBOM付きUTF-8のCSVファイルとしてダウンロード。ラベル、着地座標、パラメータ、飛行時間、海陸判定の14カラムを出力。
- **JSONエクスポート** (`exportEnsembleJSON()`): メタデータ（平均着地点、最大偏差、海陸割合等）付きのJSONとしてダウンロード。
- グローバル変数 `_ensembleResults` に各バリアントの詳細データを自動保存。

- **KDE補間**: ポイント数が少ない場合（13バリアント等）は、Box-Muller変換によるガウスカーネル密度推定（KDE）で仮想ポイントを生成し、滑らかなヒートマップを描画。
- **トグルボタン**: 「🔥 ヒートマップ」ボタンで表示/非表示を切替。
- **グラデーション**: 青→シアン→緑→黄→赤 の5段階グラデーション。

## 4. 使用方法 (How to Run)

本ツールは静的サイトとして動作しますが、ブラウザのセキュリティ制限（CORS等）を回避し、正しく動作させるためにローカルサーバー経由での実行を推奨します。

### 推奨手順 (Node.js CORSプロキシサーバー)
DockerのローカルTawhiri APIを使う場合、CORS問題を回避するために **`cors-proxy.js`** を使用してください。

```bash
# Node.jsでプロキシサーバーを起動
node cors-proxy.js
# ブラウザで http://localhost:3000 を開く
```

- ポート3000で静的ファイル配信 + `/api/` へのリクエストをDocker (port 8000) にプロキシ
- 環境変数でポート変更可能: `PORT`, `TAWHIRI_HOST`, `TAWHIRI_PORT`

### 代替手順
1. VS Code 拡張機能 "Live Server" を使用。ただし、DockerへのリクエストはCORSによりブロックされます。
2. Python: `python test.py` (ポート8000で起動)

## 5. 課題B：過去データシミュレーション環境構築手順

過去の日時（数週間前など）でシミュレーションを行う場合、SondeHubの公開APIはデータを持っていないため、ローカル環境にTawhiriサーバーを構築する必要があります。

### 前提条件
- Windows + WSL2 (Ubuntu)
- Docker Desktop

### 手順概要
詳細なコマンドは `assignment_info.txt` (またはご提供いただいた課題Bの資料) を参照してください。

1. **Dockerコンテナの準備**:
   - `ghcr.io/projecthorus/tawhiri-downloader-container:latest`
   - `ghcr.io/projecthorus/tawhiri-container:latest`

2. **過去データのダウンロード**:
   - ダウンローダを使用して、必要な日時のGFSデータセット（例: `2024091800`）を取得します。
   ```bash
   # 例: 2024年9月18日 00Z のデータを取得
   docker run --rm ... tawhiri-downloader ... 2024091800
   ```

3. **ローカルAPIサーバーの起動**:
   - ダウンロードしたデータセットをマウントして、Tawhiri APIサーバーを起動します。
   ```bash
    docker run --name tawhiri-api --rm -p 8080:8000 -v ... ghcr.io/projecthorus/tawhiri-container:latest
   ```

4. **Predictorの起動**:
   - `node cors-proxy.js` を実行し、ブラウザで `http://localhost:3000` を開きます。
   - **Launch Card** 内の「API接続先」で **「Localhost (Docker)」** を選択します。
   - 打ち上げ日時を、ダウンロードしたデータセットに対応する日時（例: 2024/09/18 ...)に設定します。
   - 「Run Prediction」を実行します。

## 6. その他・既知の問題

- **API依存**: 公開API使用時はSondeHubの稼働状況に依存します。ローカルAPI使用時は、ダウンロード済みのデータセット範囲外の日時を指定するとエラーになります。
- **実測ログフォーマット**: 現在は単純な `lat,lon` または `lat,lon,alt` のCSVに対応しています。ヘッダーは自動検知（数値でない行はスキップ）しますが、複雑なフォーマットには対応していません。
- **GeoJSON精度**: `land_japan_raw.geojson` は簡略化された陸地ポリゴンのため、離島や複雑な海岸線では精度が低下する可能性があります。沿岸部ではAPIフォールバックで補完されます。
- **Service Worker**: キャッシュバージョンは現在 `predictor-cache-v9` / `predictor-tiles-v9` です。コード変更時は `sw.js` のバージョン番号を必ずインクリメントしてください。

## 7. VPS移行時の注意点

将来的に `cors-proxy.js` をVPS等の中央サーバーへ移行する場合:

- **環境変数で設定切替**: `TAWHIRI_HOST`, `TAWHIRI_PORT`, `PORT` を環境変数で変更可能。
- **リバースプロキシ (nginx/Caddy)**: 本番環境では nginx や Caddy でリバースプロキシを設定し、SSL終端、レートリミットを追加。
- **`pred-new.js` の変更不要**: APIパスが `/api/v1/` の相対パスなので、デプロイ先が変わってもJS側の修正は不要。
- **Docker on VPS**: VPS上でも同様にTawhiri Dockerを起動し、`TAWHIRI_HOST=localhost TAWHIRI_PORT=8080` で接続。
